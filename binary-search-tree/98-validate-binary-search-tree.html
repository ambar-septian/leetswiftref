Validate Binary Search Tree

#98MediumDFSLeetCode ↗

Validate Binary Search Tree

TreeDepth-First SearchBinary Search TreeBinary Tree
01

Problem Overview

Given the root of a binary tree, determine if it is a valid binary search tree (BST).

A valid BST is defined as follows:

  • The left subtree of a node contains only nodes with keys strictly less than the node's key.
  • The right subtree of a node contains only nodes with keys strictly greater than the node's key.
  • Both the left and right subtrees must also be binary search trees.

Note: comparing only with the immediate parent is not sufficient. Every node in the left subtree must be less than the root of that subtree, not just its direct parent.

Constraints

  • The number of nodes in the tree is in the range [1, 10^4].
  • -2^31 <= Node.val <= 2^31 - 1
Key Constraint

Node values can reach the full Int32 range (-2^31 to 2^31 - 1). Initial bounds must use Int.min / Int.max (64-bit) to avoid false failures on nodes at the extremes. The BST property requires strict inequality — duplicate values make a tree invalid.

Core Insight

Pass valid (min, max) bounds down the tree. At each node verify that its value is strictly within those bounds. Recurse left with the current node's value as the new upper bound; recurse right with it as the new lower bound.

Rule of Thumb

BST validation requires global ancestor constraints, not just local parent-child checks. Whenever a problem asks whether a tree satisfies a structural invariant at every node, pass inherited bounds down the DFS rather than comparing with the immediate parent.

04

Approach

Core Insight

Pass inherited (min, max) bounds down every recursive call. At each node verify root.val is strictly within the bounds. Recurse left with the current node's value as the new upper bound and right with it as the new lower bound. Start with (Int.min, Int.max) at the root.

Define isValid(_ root: TreeNode?, left: Int, right: Int). If root is nil, return true. Check root.val > left && root.val < right — if either fails, return false. Recurse: left child with (left, root.val) and right child with (root.val, right). Initial call: isValid(root, left: Int.min, right: Int.max).

Why parent-only comparison fails

For a tree [10, 5, 15, null, null, 6, 20]: node 6 is the left child of 15 and 6 less than 15 — looks valid at the local level. But 6 is also in the right subtree of root 10, so it must be greater than 10. 6 is not greater than 10, so the tree is invalid. The bounds-passing approach catches this; the parent-only approach does not.

06

Step-by-Step Solution

Base case —

If root is nil, return true — an empty subtree is always valid.

Bounds check —

If root.val <= left or root.val >= right, return false.

Recurse with narrowed bounds —

Return isValid(root.left, left, root.val) && isValid(root.right, root.val, right).

Entry call —

isValid(root, left: Int.min, right: Int.max).

07

Code Implementation

class Solution {
  func isValidBST(_ root: TreeNode?) -> Bool {
    func isValid(_ root: TreeNode?, left: Int, right: Int) -> Bool {
      guard let root else { return true }
      guard root.val > left, root.val < right else { return false }
      return isValid(root.left,  left: left,      right: root.val)
          && isValid(root.right, left: root.val,  right: right)
    }
    return isValid(root, left: Int.min, right: Int.max)
  }
}
Swift Notes

The nested isValid function captures root from the outer scope but takes its own parameters — a clean way to add helper arguments without exposing them in the public API. guard root.val > left, root.val < right else { return false } chains two conditions in one guard with a comma, equivalent to && but more idiomatic Swift.

08

Complexity Analysis

Time Complexity — O(n)

O(n) — each node is visited exactly once. Short-circuit &&-evaluation means the first invalid node terminates the DFS immediately, making typical runtime much lower than the worst case.

Space Complexity — O(h)

O(h) for the recursion call stack where h is the tree height. O(n) for a skewed tree (degenerate linked list), O(log n) for a balanced tree.

09

Common Pitfalls

✕01
Comparing only with the direct parent
Checking root.val > root.left.val && root.val < root.right.val is insufficient. A node deep in the right subtree must be greater than the tree's root, not just its own parent.
Fix: Pass inherited bounds down the tree. Every node must satisfy the constraints imposed by all its ancestors.
✕02
Using Int.min / Int.max when node values can equal those bounds
The constraints allow node values up to 2^31 - 1 = Int32.max. If using Int32 bounds, a node with value Int32.max would incorrectly fail the strict less-than check.
Fix: Use Int.min and Int.max (64-bit) as the initial bounds, or use optional bounds (nil meaning unbounded).
✕03
Using <= instead of < in the bounds check
BST property requires strictly less than and strictly greater than — equal values are not allowed. Using <= would incorrectly accept trees with duplicate values.
Fix: Guard with root.val > left && root.val < right (strict inequalities).
