Kth Smallest Element in a BST

#230MediumIterative Inorder TraversalLeetCode ↗

Kth Smallest Element in a BST

TreeDepth-First SearchBinary Search TreeBinary Tree
01

Problem Overview

Given the root of a binary search tree and an integer k, return the kth smallest value (1-indexed) among all the values of the nodes in the tree.

A BST's inorder traversal yields values in ascending sorted order. The kth node visited during inorder traversal is the kth smallest.

Constraints:

  • The number of nodes in the tree is n.
  • 1 <= k <= n <= 10^4
  • 0 <= Node.val <= 10^4
Key Constraint

1 <= k <= n is guaranteed — a valid answer always exists. The iterative approach can return immediately on the kth node, making it more efficient than collecting all values, especially when k is small relative to n.

02

Examples

Example 1​

Input: root = [3,1,4,null,2], k = 1
Output: 1

Example 2​

Input: root = [5,3,6,2,4,null,null,1], k = 3
Output: 3

Rule of Thumb

When a BST problem requires the kth smallest (or largest) element, reach for iterative inorder traversal with an explicit stack — it visits nodes in ascending order and can stop the moment n == k without continuing the traversal.

04

Approach

Core Insight

Push the entire left spine onto a stack. Pop the top, increment a counter; when the counter equals k, return that node's value. Otherwise move to the popped node's right child and repeat — pushing its left spine. This mimics inorder traversal (left → root → right) without full recursion and stops at exactly the kth node.

Push all left-spine nodes onto a stack. Pop the top, increment a counter; when the counter equals k, return the popped node's value. Otherwise, move to the popped node's right child and repeat — pushing its left spine.

Why iterative over recursive

Recursive inorder carries the entire call stack until it returns. The iterative version can return the moment n == k without unwinding — cleaner early exit and potentially lower peak stack depth.

Step-by-Step Solution

Initialise stack and curr —

var stack: [TreeNode?] = [], var curr: TreeNode? = root, var n = 0.

Push left spine —

While curr != nil: push curr onto the stack, move curr = curr?.left.

Pop and count —

Pop the top node, increment n. If n == k, return the node's value.

Move to right subtree —

Set curr = poppedNode?.right and repeat the left-spine push from there.

07

Code Implementation

class Solution {
  func kthSmallest(_ root: TreeNode?, _ k: Int) -> Int {
    var stack: [TreeNode?] = []
    var curr: TreeNode? = root
    var n = 0

    while !stack.isEmpty || curr != nil {
      while curr != nil {
        stack.append(curr)
        curr = curr?.left
      }
      curr = stack.removeLast()
      n += 1
      if n == k { return curr!.val }
      curr = curr?.right
    }

    return -1  // unreachable given valid k
  }
}
Swift Notes

stack.removeLast() on a Swift Array is O(1) — no element shifting. This is why an array-as-stack is efficient here, unlike removeFirst() which is O(n). The while !stack.isEmpty || curr != nil condition correctly handles both the initial left-spine descent and subsequent right-subtree traversals.

08

Complexity Analysis

Time Complexity — O(H + k)

The initial left-spine push is O(H). After that, k pop-and-advance operations are performed, each O(1). Total O(H + k) — much better than O(n) when k is small or the tree is balanced (H = log n).

Space Complexity — O(H)

The explicit stack holds the current left spine at most — O(H) nodes. H = log n for a balanced BST, O(n) for a skewed one.

When to prefer Iterative vs Recursive vs Collect All

Prefer Iterative Inorder for its clean early exit at the kth node — the best approach for large trees with small k. Use Recursive Inorder when the recursive style is clearer, but note that early exit requires threading a mutable flag through the call stack. Use Collect All only for tiny trees where simplicity outweighs the wasted traversal.

09

Common Pitfalls

✕01
Incrementing n before popping instead of after
The counter represents how many nodes have been visited. It should increment when a node is popped (visited), not before.
Fix: curr = stack.removeLast(); n += 1; if n == k { return curr!.val } — pop, count, check.
✕02
Not moving to the right subtree after popping
After processing a node, the next inorder node is the leftmost node in its right subtree. Failing to set curr = curr?.right after the pop means the right subtree is never visited.
Fix: Always set curr = poppedNode?.right as the last step in the outer loop body.
✕03
Returning n instead of the node's value
The counter n is the ordinal position; the answer is the actual node value curr!.val.
Fix: Return curr!.val when n == k, not n.
