Find Peak Element

ArrayBinary Search
01

Problem Overview

A peak element is an element that is strictly greater than its neighbors.

Given a 0-indexed integer array nums, find a peak element and return its index. If the array contains multiple peaks, return the index to any of the peaks.

You may imagine that nums[-1] = nums[n] = -∞. In other words, an element is always considered to be strictly greater than a neighbor that is out of the boundary.

You must write an algorithm that runs in O(log n) time.

Constraints:

  • 1 <= nums.length <= 1000
  • -2^31 <= nums[i] <= 2^31 - 1
  • nums[i] != nums[i + 1] for all valid i (no equal adjacent elements).
02

Examples

Example 1​

Input: nums = [1,2,3,1]
Output: 2
Explanation: nums[2]=3 is a peak (3 > 2 and 3 > 1).

Example 2​

Input: nums = [1,2,1,3,5,6,4]
Output: 5
Explanation: Either index 1 or 5 is acceptable (nums[1]=2 and nums[5]=6 are both peaks).
03

Pattern Recognition

→ Pattern Decision Tree
?
Does the problem require O(log n)?
✓ Yes — A linear scan finds a peak in O(n) but the constraint demands O(log n) — binary search required.
?
Can we determine which half contains a peak at each midpoint?
✓ Yes — If nums[m] less than nums[m+1], a peak exists to the right (ascending slope must peak somewhere). If nums[m] less than nums[m-1], a peak exists to the left.
?
Is there always at least one peak?
✓ Yes — Boundaries are -inf. Any local maximum qualifies. Even a sorted array has a peak at its last element.
Tap to reveal pattern →
04

Approach

At each midpoint m, check which direction is ascending:

  • If nums[m] < nums[m-1] — the left side has a higher neighbor, so a peak exists there: r = m - 1.
  • If nums[m] < nums[m+1] — the right side is ascending, so a peak exists there: l = m + 1.
  • Otherwise nums[m] is locally greater than both neighbors — it is a peak: return m.
Why moving toward the higher neighbor guarantees a peak

An ascending slope must eventually peak because the array is finite and boundaries are -inf. Moving toward the higher neighbor always keeps at least one peak in the remaining search range.

06

Step-by-Step Solution

Handle single-element array —

If nums.count == 1, return 0 immediately (only element is a peak by boundary rule).

Binary search loop —

while l <= r: compute m = l + (r - l) / 2.

Check left neighbor —

If m > 0 and nums[m] < nums[m-1]: set r = m - 1.

Check right neighbor —

Else if m < count - 1 and nums[m] < nums[m+1]: set l = m + 1.

Peak found —

Else: nums[m] is greater than both neighbors — return m.

07

Code Implementation

class Solution {
  func findPeakElement(_ nums: [Int]) -> Int {
    let c = nums.count
    guard c > 1 else { return 0 }

    var l = 0
    var r = c - 1

    while l <= r {
      let m = l + (r - l) / 2

      if m > 0, nums[m] < nums[m - 1] {
        r = m - 1
      } else if m < c - 1, nums[m] < nums[m + 1] {
        l = m + 1
      } else {
        return m
      }
    }

    return 0  // unreachable given valid input
  }
}

09

Common Pitfalls

✕01
Checking only one neighbor
Checking only the right neighbor misses cases where the left neighbor is larger, causing the search to move right when it should go left.
Fix: Check the left neighbor first (if m > 0 and nums[m] less than nums[m-1]), then check the right neighbor. Both conditions must be evaluated.
✕02
Out-of-bounds access on boundary midpoints
When m=0, accessing nums[m-1] causes an index out-of-bounds error.
Fix: Guard with m > 0 before checking the left neighbor and m less than count - 1 before the right neighbor.
✕03
Setting r = m instead of r = m - 1
When moving left, setting r = m keeps the midpoint in the search space, potentially creating an infinite loop when l == m.
Fix: Use r = m - 1 to exclude the current midpoint from the left half search.
