<SectionHeader n={1} title="Problem Overview" />

Given the `root` of a binary tree, imagine yourself standing on the **right side** of it. Return the values of the nodes you can see ordered from top to bottom.

A node is visible from the right side if it is the **rightmost node at its level** — there is no other node at the same depth that would block it when viewed from the right.

**Constraints:**
- The number of nodes in the tree is in the range `[0, 100]`.
- `-100 <= Node.val <= 100`

<Callout type="amber" label="Key Constraint">
The number of nodes can be 0 (empty tree). Returning `[]` immediately when `root` is `nil` — before any BFS or DFS traversal — is a required edge-case guard.
</Callout>

<SectionHeader n={2} title="Examples" />

#### Example 1

```
Input: root = [1,2,3,null,5,null,4]
Output: [1,3,4]
```

#### Example 2

```
Input: root = [1,null,3]
Output: [1,3]
```

#### Example 3

```
Input: root = []
Output: []
```

<Callout type="green" label="Rule of Thumb">
When a problem asks for one representative node per level of a binary tree, BFS with a level-width snapshot is the most direct mapping — the last (or first) node dequeued per level is the answer.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
Process the tree level by level using BFS. Snapshot the queue size before the inner loop so that children added during processing do not blur the level boundary. The last node dequeued per level is the rightmost visible node — append its value to the result.
</Callout>

<RightSideViewDiagram />

Use a queue initialised with the root. At each iteration, snapshot `count = queue.count` (the width of the current level). Process exactly `count` nodes: enqueue their children, and record the value of the last one (`i == count - 1`). Repeat until the queue is empty.

<Callout type="cyan" label="Why snapshot count first">
Children are added to the queue during the same loop. Snapshotting <code>count</code> before the loop prevents mixing the current level's children into the current level's processing.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Guard nil root">
    Return `[]` immediately if `root` is nil.
  </StepItem>
  <StepItem title="Initialise queue">
    `var queue: [TreeNode?] = [root]`, `var result = [Int]()`.
  </StepItem>
  <StepItem title="BFS loop">
    While `queue` is not empty: snapshot `let count = queue.count`.
  </StepItem>
  <StepItem title="Process level">
    For `i in 0..count`: dequeue `item`. If `i == count - 1`, append `item.val` to `result`. Enqueue left and right children if non-nil.
  </StepItem>
  <StepItem title="Return result">
    Return `result` after the BFS loop.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func rightSideView(_ root: TreeNode?) -> [Int] {
    guard let root else { return [] }
    var queue: [TreeNode?] = [root]
    var result = [Int]()

    while !queue.isEmpty {
      let count = queue.count

      for i in 0..<count {
        guard let item = queue.removeFirst() else { continue }
        if i == count - 1 {
          result.append(item.val)
        }
        if let left  = item.left  { queue.append(left)  }
        if let right = item.right { queue.append(right) }
      }
    }

    return result
  }
}
```

<Callout type="green" label="Swift Notes">
`queue.removeFirst()` on a Swift `Array` is O(n) because it shifts all elements. For production code with large trees, prefer a `Deque` from the swift-collections package (O(1) front removal). In interview contexts with the given constraint (at most 100 nodes), the `Array` approach is acceptable.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(n)">
Every node is enqueued and dequeued exactly once. The inner loop at each level does O(1) work per node. Total work is O(n) where n is the number of nodes.
</Callout>

<Callout type="green" label="Space Complexity — O(n)">
The queue holds at most one full level at a time. In a complete binary tree the widest level has n/2 nodes, so queue space is O(n). The result array holds one value per level, O(h) ≤ O(n).
</Callout>

<Callout type="amber" label="When to prefer BFS vs DFS">
Prefer **BFS** when you need level-order information naturally (e.g., also tracking level indices or sums) — the level grouping is explicit. Prefer **DFS (right-first)** when call-stack depth is not a concern and you want to avoid the queue; it uses O(h) space vs O(n) for BFS on wide trees.
</Callout>
