<SectionHeader n={1} title="Problem Overview" />

Roman numerals are represented by seven symbols:

| Symbol | Value |
|--------|-------|
| I | 1 |
| V | 5 |
| X | 10 |
| L | 50 |
| C | 100 |
| D | 500 |
| M | 1000 |

Roman numerals are usually written largest to smallest from left to right. However, there are six subtractive cases:
- `IV` = 4, `IX` = 9
- `XL` = 40, `XC` = 90
- `CD` = 400, `CM` = 900

Given a roman numeral string `s`, convert it to an integer.

**Constraints**
- `1 <= s.length <= 15`
- `s` contains only the characters `I`, `V`, `X`, `L`, `C`, `D`, `M`.
- It is guaranteed that `s` is a valid Roman numeral in the range `[1, 3999]`.

<Callout type="amber" label="Key Constraint">
Only six subtractive cases exist (IV, IX, XL, XC, CD, CM) and input is always valid — no need to validate the numeral. All other symbol sequences are purely additive from left to right.
</Callout>

<Callout type="green" label="Core Insight">
Scan left to right, tracking the previous value. When the current value is greater than the previous, a subtractive pair has just been completed — the previous value was added once too many, so subtract it twice to correct. Otherwise just add the current value.
</Callout>

<SectionHeader n={2} title="Examples" />

#### Example 1
```
Input:  s = "III"
Output: 3
Explanation: III = 1 + 1 + 1 = 3
```

#### Example 2
```
Input:  s = "LVIII"
Output: 58
Explanation: L = 50, V = 5, III = 3 → 58
```

#### Example 3
```
Input:  s = "MCMXCIV"
Output: 1994
Explanation: M = 1000, CM = 900, XC = 90, IV = 4 → 1994
```

When parsing Roman numerals, think lookbehind — if the current symbol is larger than the previous one, a subtractive pair just completed; subtract the previous value twice to correct the over-addition.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
Scan left to right, adding each symbol's value optimistically. When a larger symbol follows a smaller one, the smaller was a subtractive prefix — subtract it twice (once to undo the over-addition, once for the correct subtraction). This single lookbehind check handles all six subtractive cases with no lookahead required.
</Callout>

Iterate through each character, mapping it to its integer value via a switch. Maintain `temp` (the previous character's value). If `current > temp` and `temp != 0`, a subtractive pair is complete: add `current - 2 * temp` (effectively undoing the +temp from the previous step and applying the correct subtraction). Otherwise just add `current`. Update `temp` and continue.

<Callout type="cyan" label="Why subtract twice">
When we processed the previous character, we added its value to the result optimistically. Now we learn it should have been subtracted. We need to remove it (+temp → 0) and then subtract it (-temp), which is -2*temp net. So: result += current - 2*temp.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Map each character to its value">
    Use a switch on the current character to get its integer value (I=1, V=5, X=10, L=50, C=100, D=500, M=1000).
  </StepItem>
  <StepItem title="Check for subtractive pair">
    If `current > temp` and `temp != 0`, add `current - 2 * temp` to result to correct the over-addition from the previous step.
  </StepItem>
  <StepItem title="Normal addition">
    Otherwise add `current` to result directly.
  </StepItem>
  <StepItem title="Update previous value">
    Set `temp = current` and continue to the next character.
  </StepItem>
  <StepItem title="Return result">
    After processing all characters, `result` holds the integer value.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func romanToInt(_ s: String) -> Int {
    var result = 0
    var temp = 0

    for char in s {
      let number: Int
      switch char {
      case "I": number = 1
      case "V": number = 5
      case "X": number = 10
      case "L": number = 50
      case "C": number = 100
      case "D": number = 500
      case "M": number = 1000
      default:  number = 0
      }

      if number > temp, temp != 0 {
        result += (number - (temp * 2))
      } else {
        result += number
      }

      temp = number
    }

    return result
  }
}
```

<Callout type="green" label="Swift Notes">
Swift `switch` on a `Character` must be exhaustive — the `default: number = 0` case is required even though the problem guarantees only valid symbols. Declaring `let number: Int` inside the switch then using it after keeps the mapping and the logic cleanly separated.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(n)">
Each character is processed exactly once with O(1) work per character. With at most 15 characters the input size is bounded, but the algorithm is O(n) in the general case.
</Callout>

<Callout type="green" label="Space Complexity — O(1)">
Only two integer variables — result and temp — are maintained. No auxiliary data structures are needed.
</Callout>
