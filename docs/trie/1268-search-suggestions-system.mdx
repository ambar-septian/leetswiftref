<SectionHeader n={1} title="Problem Overview" />

You are given an array of strings `products` and a string `searchWord`.

Design a system that suggests at most three product names from `products` after each character of `searchWord` is typed. Suggested products should have a common prefix with `searchWord`. If there are more than three products with a common prefix, return the three lexicographically minimum products.

Return a list of lists of the suggested products after each character of `searchWord` is typed.

**Constraints:**
- `1 <= products.length <= 1000`
- `1 <= products[i].length <= 3000`
- `1 <= sum(products[i].length) <= 2 * 10^4`
- All the strings of `products` are unique.
- `1 <= searchWord.length <= 1000`
- `products[i]` and `searchWord` consist of lowercase English letters only.

<Callout type="amber" label="Key Constraint">
  At most 3 suggestions are returned per prefix — always the 3 lexicographically smallest. Once the prefix is absent from the trie (a character is missing), all subsequent prefixes must return an empty list.
</Callout>

<SectionHeader n={2} title="Examples" />

#### Example 1

```
Input:  products = ["mobile","mouse","moneypot","monitor","mousepad"],
        searchWord = "mouse"
Output: [
  ["mobile","moneypot","monitor"],
  ["mobile","moneypot","monitor"],
  ["mouse","mousepad"],
  ["mouse","mousepad"],
  ["mouse","mousepad"]
]
```

#### Example 2

```
Input:  products = ["havana"], searchWord = "havana"
Output: [["havana"],["havana"],["havana"],["havana"],["havana"],["havana"]]
```

<Callout type="green" label="Rule of Thumb">
  When you need the top-k lexicographically smallest matches for every prefix of a query, sort the input first and build a trie that caches up to k results at each node during insertion — lookup then becomes a simple traversal with no per-query sorting.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
  Sort products before insertion so the first 3 products appended to any trie node are already the lexicographic minimum — no sorting during lookup is needed. During query, follow the searchWord characters through the trie; the current node's cached results array is the answer for that prefix. A nil node means the prefix is absent — all remaining results are empty.
</Callout>

Sort `products` lexicographically first. Then insert each product into the trie one character at a time. At every node along the path, call `appendResult(product)` — since products are inserted in sorted order, each node naturally accumulates the first (smallest) 3 that share its prefix.

For the query phase, traverse the trie following `searchWord` characters. At each step, the current node's `results` array is the answer for that prefix. If the trie path breaks (a character is missing), all remaining prefixes return an empty list.

<Callout type="green" label="Why sort first?">
Sorting before insertion guarantees that `appendResult` fills each node's results array with the lexicographically smallest products — no sorting is needed during lookups.
</Callout>

<Callout type="cyan" label="Guard on appendResult">
`appendResult` only appends when `results.count less than 3`. This caps memory and ensures O(1) per append — no priority queue needed.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Sort products">
    Sort `products` lexicographically so that insertion order determines lexicographic priority.
  </StepItem>
  <StepItem title="Build trie">
    For each product, walk the trie character by character, creating nodes as needed. At each node, call `appendResult(product)` to store it if fewer than 3 results are already cached.
  </StepItem>
  <StepItem title="Query searchWord">
    Traverse from the root following each character of `searchWord`. At each step, append the current node's `results` array (or `[]` if the path breaks) to the output.
  </StepItem>
  <StepItem title="Handle missing prefix">
    If any character is not found in the trie (`curr?.children[char]` is nil), set `curr = nil`. Subsequent iterations will always produce `[]` via the `?? []` fallback.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  let root = Node()

  func suggestedProducts(_ products: [String], _ searchWord: String) -> [[String]] {
    let sortedProducts = products.sorted()

    for product in sortedProducts {
      var current = root
      for char in product {
        if current.children[char] == nil {
          current.children[char] = Node()
        }
        current = current.children[char]!
        current.appendResult(product)
      }
    }

    var results = [[String]]()
    var curr: Node? = root
    for char in searchWord {
      let products = curr?.children[char]?.results ?? []
      results.append(products)
      curr = curr?.children[char]
    }

    return results
  }
}

class Node {
  var children = [Character: Node]()
  private(set) var results = [String]()

  func appendResult(_ string: String) {
    guard results.count < 3 else { return }
    results.append(string)
  }
}
```

<Callout type="green" label="Swift Notes">
  `var children = [Character: Node]()` uses a dictionary for trie children — O(1) average access. `private(set) var results` exposes read access while restricting writes to `appendResult`. Optional chaining `curr?.children[char]?.results ?? []` cleanly handles a nil node without a crash — once `curr` becomes nil, all subsequent lookups return `[]` naturally.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(M log M + M + N)">
  Sorting products takes O(M log M) where M = total characters across all products. Trie insertion is O(M) — each character creates or traverses one node. Query traversal is O(N) where N = searchWord length. Each node lookup is O(1).
</Callout>

<Callout type="green" label="Space Complexity — O(M)">
  The trie stores at most M nodes (one per character across all products). Each node holds a dictionary of children and up to 3 result strings. The result strings themselves take O(M) total space. Query uses O(1) auxiliary space beyond the output.
</Callout>
