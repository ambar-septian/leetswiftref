<SectionHeader n={1} title="Problem Overview" />

Given an array of **distinct** integers `candidates` and a target integer `target`, return a list of all **unique combinations** of `candidates` where the chosen numbers sum to `target`. You may return the combinations in any order.

The **same** number may be chosen from `candidates` an **unlimited number of times**. Two combinations are unique if the frequency of at least one of the chosen numbers is different.

The test cases are generated such that the number of unique combinations that sum up to `target` is less than `150` combinations for the given input.

**Constraints**
- `1 <= candidates.length <= 30`
- `2 <= candidates[i] <= 40`
- All elements of `candidates` are **distinct**.
- `1 <= target <= 40`

<Callout type="amber" label="Key Constraint">
Candidates are distinct and each can be reused an unlimited number of times. All candidates are positive (≥ 2), so a running sum that exceeds the target can never recover — prune immediately.
</Callout>

<Callout type="green" label="Core Insight">
Use DFS starting from a `startIndex` that never moves backwards. Recurse with `dfs(i)` — not `dfs(i+1)` — to allow reusing the current element. Prune immediately when the running sum exceeds the target.
</Callout>

<SectionHeader n={2} title="Examples" />

#### Example 1
```
Input:  candidates = [2,3,6,7], target = 7
Output: [[2,2,3],[7]]
```

#### Example 2
```
Input:  candidates = [2,3], target = 6
Output: [[2,2,2],[3,3]]
```

#### Example 3
```
Input:  candidates = [2], target = 1
Output: []
```

<Callout type="green" label="Rule of Thumb">
When combinations allow unlimited element reuse, think backtracking with `dfs(i)` (not i+1) to permit reuse, plus a startIndex to ensure elements are picked in non-decreasing order so [2,3] and [3,2] are not both generated.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
Use a startIndex so each recursive call only considers candidates from the current index onward — this ensures combinations are always built in non-decreasing index order, preventing [2,3] and [3,2] from both appearing. Pass dfs(i) rather than dfs(i+1) to allow reusing the same element. Prune immediately when the running sum exceeds the target.
</Callout>

Maintain a shared `paths` array and a `sums` running total. The recursive function `dfs(_ startIndex)` iterates from `startIndex` to the end of `candidates`. For each candidate `c`: append it, add to `sums`, recurse with `dfs(i)` (same index — allows reuse), then remove and subtract (backtrack). Base cases: `sums == target` → record result; `sums > target` → prune.

<CombinationSumDiagram />

<Callout type="cyan" label="Why startIndex prevents duplicate combinations">
Without a startIndex every DFS call could pick any element, generating [2,3] and [3,2] as separate results. Starting each recursive call from index `i` (not 0) ensures elements are always added in non-decreasing index order, so each combination appears exactly once.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Initialise shared state">
    `results = [[Int]]()`, `paths = [Int]()`, `sums = 0`.
  </StepItem>
  <StepItem title="Base cases">
    `sums == target` → append a copy of `paths` and return. `sums > target` → return immediately (prune).
  </StepItem>
  <StepItem title="Iterate from startIndex">
    For `i` from `startIndex` to `candidates.count - 1`: append `candidates[i]`, add to `sums`, call `dfs(i)` (same i — reuse allowed).
  </StepItem>
  <StepItem title="Backtrack">
    After each recursive call: `paths.removeLast()`, `sums -= candidates[i]`.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func combinationSum(_ candidates: [Int], _ target: Int) -> [[Int]] {
    var results = [[Int]]()
    var paths = [Int]()
    var sums = 0

    func dfs(_ startIndex: Int) {
      if sums == target {
        results.append(paths)
        return
      } else if sums > target {
        return
      }
      for i in startIndex..<candidates.count {
        paths.append(candidates[i])
        sums += candidates[i]
        dfs(i)            // i, not i+1 — reuse allowed
        paths.removeLast()
        sums -= candidates[i]
      }
    }

    dfs(0)
    return results
  }
}
```

<Callout type="green" label="Swift Notes">
Swift Array is a value type, so `results.append(paths)` copies the current path automatically — no `Array(paths)` wrapping is needed unlike reference-type languages. The nested `func dfs` captures `paths`, `sums`, `results`, and `candidates` by reference from the enclosing function scope.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(N^(T/M))">
The recursion tree has branching factor N and maximum depth T/M (where T is target and M is the minimum candidate value). Upper bound on nodes explored is O(N^(T/M)), each doing O(1) work.
</Callout>

<Callout type="green" label="Space Complexity — O(T/M)">
The recursion stack depth is at most T/M levels. The path array holds at most T/M elements at any time. Space is O(T/M).
</Callout>
