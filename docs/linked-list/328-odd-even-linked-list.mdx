<SectionHeader n={1} title="Problem Overview" />

Given the `head` of a singly linked list, group all the nodes with odd indices together followed by the nodes with even indices, and return the reordered list.

The **first** node is considered odd, the **second** even, and so on. Note that the relative order inside both groups should be preserved.

You must solve the problem in O(1) extra space complexity and O(n) time complexity.

**Constraints:**
- The number of nodes in the linked list is in the range `[0, 10^4]`.
- `-10^6 <= Node.val <= 10^6`

<Callout type="amber" label="Key Constraint">
O(1) extra space and O(n) time are required. This rules out collecting odd/even nodes into separate arrays. The only O(1) approach is in-place pointer weaving — connecting each odd node directly to the next odd node and each even node to the next even node.
</Callout>

<Callout type="green" label="Rule of Thumb">
When separating a linked list into two groups in O(1) space, weave two chains simultaneously in a single loop, then join them. Save the even-chain head before the loop — it is overwritten on the first iteration.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
`odd` and `even` pointers walk their respective chains simultaneously. Each iteration: set `odd.next = even.next` (skip the even node), then set `even.next = odd.next.next` (skip the new odd node). Both pointers then advance. Saving `evenHead` before the loop preserves the entry point needed to join the two chains at the end.
</Callout>

`odd` starts at `head`, `even` starts at `head.next`, `evenHead = even`. Each iteration:
- `odd.next = even.next` (skip even node in odd chain)
- `even.next = odd.next.next` (skip odd node in even chain, i.e. the new odd.next's next)
- Advance `odd = odd.next`, `even = even.next`

After the loop, `odd.next = evenHead` to join the chains.

<Callout type="cyan" label="Loop termination condition">
Both <code>odd?.next != nil</code> AND <code>even?.next != nil</code> must hold — if either pointer's next is nil, all remaining links are already set correctly and we can exit.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Save even head">
    `var evenHead = head?.next`. This anchor is needed to join the chains at the end.
  </StepItem>
  <StepItem title="Weave chains in the loop">
    While `odd?.next != nil && even?.next != nil`: redirect `odd.next` to skip the even node, redirect `even.next` to skip the new odd node, then advance both pointers.
  </StepItem>
  <StepItem title="Join odd and even chains">
    `odd?.next = evenHead`.
  </StepItem>
  <StepItem title="Return head">
    The head of the odd chain is the original `head`.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func oddEvenList(_ head: ListNode?) -> ListNode? {
    var odd     = head
    var even    = head?.next
    var evenHead = even

    while odd?.next != nil, even?.next != nil {
      odd?.next  = even?.next
      even?.next = odd?.next?.next
      odd  = odd?.next
      even = even?.next
    }

    odd?.next = evenHead
    return head
  }
}
```

<Callout type="green" label="Swift Notes">
Optional chaining (`odd?.next`, `even?.next`) makes pointer operations safe for short lists without explicit nil checks. The while condition uses comma-separated `&&` semantics: `while odd?.next != nil, even?.next != nil`. Note that `even?.next = odd?.next?.next` reads the new `odd.next` (already updated) — this is intentional and correct.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(n)">
A single pass through all n nodes: `odd` and `even` advance together, visiting every node exactly once. The final join (`odd?.next = evenHead`) is O(1). Total: O(n).
</Callout>

<Callout type="green" label="Space Complexity — O(1)">
Only three pointer variables (`odd`, `even`, `evenHead`) are used regardless of list length. All pointer modifications are on existing nodes — no new allocations.
</Callout>
