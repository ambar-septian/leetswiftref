<SectionHeader n={1} title="Problem Overview" />

Given the `head` of a singly linked list, return `true` if it is a palindrome or `false` otherwise.

**Constraints:**
- The number of nodes in the list is in the range `[1, 10^5]`.
- `0 <= Node.val <= 9`

<Callout type="amber" label="Key Constraint">
The follow-up asks for O(n) time and O(1) space. This rules out copying values to an array (O(n) space) and recursion (O(n) call stack). The only O(1)-space approach is in-place reversal of the second half.
</Callout>

<Callout type="green" label="Rule of Thumb">
When checking if a singly linked list is a palindrome in O(1) space, combine three techniques: fast/slow pointers to find the midpoint, in-place reversal of the second half, and simultaneous two-pointer comparison of both halves.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
Three phases: (1) fast/slow pointers advance `slow` to the last node of the first half; (2) reverse the second half in-place starting from `slow`; (3) walk `p1` from `head` and `p2` from the reversed head simultaneously — return `false` on any mismatch. `p2` exhaustion (not `p1`) determines termination.
</Callout>

Three phases:
1. **Find the midpoint** — move `slow` one step and `fast` two steps until `fast.next == nil` or `fast.next.next == nil`. `slow` ends at the last node of the first half.
2. **Reverse the second half** — iterative three-pointer reversal starting from `slow`, producing a reversed chain (`prev`).
3. **Compare both halves** — walk `p1` from `head` and `p2` from `prev` simultaneously; if any values differ, return `false`.

<Callout type="cyan" label="Odd vs even length">
For odd-length lists the middle node is skipped — only the nodes after it form the second half. The fast/slow condition <code>fast?.next != nil and fast?.next?.next != nil</code> correctly positions slow at the end of the first half in both cases.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Find the midpoint with fast/slow pointers">
    Move `slow` one step and `fast` two steps each iteration until `fast?.next == nil` or `fast?.next?.next == nil`.
  </StepItem>
  <StepItem title="Reverse the second half">
    Starting from `slow` (inclusive), apply the iterative reversal: save `next`, redirect `curr.next = prev`, advance both.
  </StepItem>
  <StepItem title="Compare first and second halves">
    Walk `p1` from `head` and `p2` from `prev`. Return `false` on any value mismatch.
  </StepItem>
  <StepItem title="Return true">
    If `p2` is exhausted without a mismatch, the list is a palindrome.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func isPalindrome(_ head: ListNode?) -> Bool {
    var slow = head
    var fast = head

    while fast?.next != nil && fast?.next?.next != nil {
      slow = slow?.next
      fast = fast?.next?.next
    }

    // Reverse second half starting from slow
    var prev: ListNode? = nil
    var curr = slow

    while curr != nil {
      let next = curr?.next
      curr?.next = prev
      prev = curr
      curr = next
    }

    // Compare both halves
    var p1 = head
    var p2 = prev

    while p1 != nil, p2 != nil {
      if p1?.val != p2?.val { return false }
      p1 = p1?.next
      p2 = p2?.next
    }

    return true
  }
}
```

<Callout type="green" label="Swift Notes">
The reversal block (`prev`, `curr`, `next` pattern) is identical to problem 206 — recognising this avoids re-deriving it from scratch. `fast?.next != nil && fast?.next?.next != nil` uses optional chaining; both conditions must be non-nil to advance, so the conjunction short-circuits safely without force-unwrapping.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(n)">
Three linear passes: O(n) to find the midpoint, O(n/2) to reverse the second half, O(n/2) to compare both halves. Total O(n).
</Callout>

<Callout type="green" label="Space Complexity — O(1)">
Only a fixed number of pointer variables (`slow`, `fast`, `prev`, `curr`, `p1`, `p2`). No extra arrays or data structures are allocated.
</Callout>

<Callout type="amber" label="When to prefer Fast/Slow+Reversal vs Array Copy vs Recursion">
Prefer **Fast/Slow + In-Place Reversal** when O(1) space is required — the only constant-space option and the expected interview answer. Use **Copy to Array** when code clarity matters more than space. Use **Recursion** only when the elegant structure is valued; the O(n) call stack risks overflow on lists up to 100,000 nodes.
</Callout>
