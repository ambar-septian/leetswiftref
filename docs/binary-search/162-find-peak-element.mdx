<SectionHeader n={1} title="Problem Overview" />

A **peak element** is an element that is strictly greater than its neighbors.

Given a 0-indexed integer array `nums`, find a peak element and return its index. If the array contains multiple peaks, return the index to **any** of the peaks.

You may imagine that `nums[-1] = nums[n] = -∞`. In other words, an element is always considered to be strictly greater than a neighbor that is out of the boundary.

You must write an algorithm that runs in **O(log n)** time.

**Constraints:**
- `1 <= nums.length <= 1000`
- `-2^31 <= nums[i] <= 2^31 - 1`
- `nums[i] != nums[i + 1]` for all valid `i` (no equal adjacent elements).

<Callout type="amber" label="Key Constraint">
No two adjacent elements are equal. This guarantees that at every midpoint, one neighbor is strictly larger — the binary search direction is always deterministic with no tie-breaking edge case.
</Callout>

<SectionHeader n={2} title="Examples" />

#### Example 1

```
Input: nums = [1,2,3,1]
Output: 2
Explanation: nums[2]=3 is a peak (3 > 2 and 3 > 1).
```

#### Example 2

```
Input: nums = [1,2,1,3,5,6,4]
Output: 5
Explanation: Either index 1 or 5 is acceptable (nums[1]=2 and nums[5]=6 are both peaks).
```

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
At any midpoint, the neighbor with the higher value guarantees a peak exists on that side — an ascending slope must crest before the array ends or hits the -∞ boundary. Moving toward the higher neighbor always keeps at least one peak inside the remaining search range.
</Callout>

At each midpoint `m`, check which direction is ascending:
- If `nums[m] < nums[m-1]` — the left side has a higher neighbor, so a peak exists there: `r = m - 1`.
- If `nums[m] < nums[m+1]` — the right side is ascending, so a peak exists there: `l = m + 1`.
- Otherwise `nums[m]` is locally greater than both neighbors — it is a peak: return `m`.

<Callout type="cyan" label="Why moving toward the higher neighbor guarantees a peak">
An ascending slope must eventually peak because the array is finite and boundaries are -inf. Moving toward the higher neighbor always keeps at least one peak in the remaining search range.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Handle single-element array">
    If `nums.count == 1`, return `0` immediately (only element is a peak by boundary rule).
  </StepItem>
  <StepItem title="Binary search loop">
    `while l <= r`: compute `m = l + (r - l) / 2`.
  </StepItem>
  <StepItem title="Check left neighbor">
    If `m > 0` and `nums[m] < nums[m-1]`: set `r = m - 1`.
  </StepItem>
  <StepItem title="Check right neighbor">
    Else if `m < count - 1` and `nums[m] < nums[m+1]`: set `l = m + 1`.
  </StepItem>
  <StepItem title="Peak found">
    Else: `nums[m]` is greater than both neighbors — return `m`.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func findPeakElement(_ nums: [Int]) -> Int {
    let c = nums.count
    guard c > 1 else { return 0 }

    var l = 0
    var r = c - 1

    while l <= r {
      let m = l + (r - l) / 2

      if m > 0, nums[m] < nums[m - 1] {
        r = m - 1
      } else if m < c - 1, nums[m] < nums[m + 1] {
        l = m + 1
      } else {
        return m
      }
    }

    return 0  // unreachable given valid input
  }
}
```

<Callout type="green" label="Swift Notes">
Using `l + (r - l) / 2` instead of `(l + r) / 2` prevents integer overflow when both `l` and `r` are near `Int.max`. The `guard c > 1 else { return 0 }` pattern short-circuits the trivial single-element case before entering the binary search loop.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(log n)">
The search space halves on every iteration: each comparison moves either `l` up or `r` down by at least 1, so the loop runs at most log₂(n) iterations. The three neighbor comparisons per iteration are O(1), giving O(log n) overall.
</Callout>

<Callout type="green" label="Space Complexity — O(1)">
Only two integer pointers (`l` and `r`) plus the midpoint `m` are allocated, regardless of array size. No auxiliary data structure is used.
</Callout>
