<SectionHeader n={1} title="Problem Overview" />

You are given two positive integer arrays `spells` and `potions`, of length `n` and `m` respectively, where `spells[i]` represents the strength of the `i`-th spell and `potions[j]` represents the strength of the `j`-th potion.

You are also given an integer `success`. A spell and potion pair is considered **successful** if the **product** of their strengths is **at least** `success`.

Return an integer array `pairs` of length `n` where `pairs[i]` is the number of **potions** that will form a successful pair with the `i`-th spell.

**Constraints:**
- `n == spells.length`
- `m == potions.length`
- `1 <= n, m <= 10^5`
- `1 <= spells[i], potions[j] <= 10^5`
- `1 <= success <= 10^10`

<Callout type="amber" label="Key Constraint">
  success can be up to 10^10, requiring 64-bit integers. Swift's Int is 64-bit on modern platforms, so spell * potion (max 10^10) fits without overflow. O(n·m) brute force = 10^10 ops — binary search is required.
</Callout>

<SectionHeader n={2} title="Examples" />

#### Example 1

```
Input:  spells = [5,1,3], potions = [1,2,3,4,5], success = 7
Output: [4,0,3]
Explanation:
  spell 5: 5*1=5<7, 5*2=10>=7 → 4 potions work (indices 1..4)
  spell 1: 1*5=5<7 → 0 potions
  spell 3: 3*3=9>=7 → 3 potions (indices 2..4)
```

#### Example 2

```
Input:  spells = [3,1,2], potions = [8,5,8], success = 16
Output: [2,0,2]
```

<Callout type="green" label="Rule of Thumb">
  When counting how many sorted elements satisfy a threshold condition per query, sort once and binary-search for the leftmost valid index — count is `length - leftmost_index`.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
  Sort potions ascending once. For each spell, binary-search for the first potion index where `spell * potion &gt;= success`. All potions from that index onward qualify, so the count is `m - l` without scanning them individually.
</Callout>

Sort `potions` ascending. For each spell, binary search for the leftmost index `l` where `spell * potions[l] >= success`. The count of valid potions is `m - l`.

Use a standard left-boundary binary search: if `power < success`, move `l = mid + 1`; otherwise move `r = mid`. After the loop, `l` is the first valid index.

<Callout type="yellow" label="Integer overflow risk">
`spell * potions[m]` can reach 10^5 * 10^5 = 10^10, which exceeds Int32 range. Swift's `Int` is 64-bit on modern platforms (up to ~9.2 * 10^18), so no overflow here — but always verify platform integer size.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Sort potions">
    `let sortedPotions = potions.sorted()`
  </StepItem>
  <StepItem title="Binary search per spell">
    For each spell, initialise `l = 0`, `r = m`. While `l < r`: compute `mid = l + (r-l)/2`. If `spell * sortedPotions[mid] < success`, set `l = mid + 1`; else set `r = mid`.
  </StepItem>
  <StepItem title="Count valid potions">
    `result[i] = m - l`. All potions from index `l` onward are valid.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

```swift
class Solution {
  func successfulPairs(_ spells: [Int], _ potions: [Int], _ success: Int) -> [Int] {
    var result = Array(repeating: 0, count: spells.count)
    let sorted = potions.sorted()

    for i in 0..<spells.count {
      let spell = spells[i]
      var l = 0, r = sorted.count

      while l < r {
        let m = l + (r - l) / 2
        if spell * sorted[m] < success { l = m + 1 }
        else { r = m }
      }

      result[i] = sorted.count - l
    }

    return result
  }
}
```

<Callout type="green" label="Swift Notes">
  `let m = l + (r - l) / 2` avoids integer overflow compared to `(l + r) / 2`. Using `r = sorted.count` (not `sorted.count - 1`) as the initial upper bound means `l` naturally lands at `sorted.count` when no potion qualifies, giving a count of 0 without a special case.
</Callout>

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O((n + m) log m)">
  Sorting potions takes O(m log m). Each of the n spells triggers one binary search over m potions costing O(log m). Total: O(m log m + n log m) = O((n + m) log m).
</Callout>

<Callout type="green" label="Space Complexity — O(m)">
  The sorted copy of potions requires O(m) extra space. The result array is O(n). The input arrays are not mutated.
</Callout>
