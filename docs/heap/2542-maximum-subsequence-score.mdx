<SectionHeader n={1} title="Problem Overview" />

You are given two **0-indexed** integer arrays `nums1` and `nums2` of equal length `n` and a positive integer `k`. You must choose a subsequence of indices of length `k`.

For chosen indices `i0, i1, ..., ik-1`, the **score** is defined as:

- The **sum** of the selected elements from `nums1`: `nums1[i0] + nums1[i1] + ... + nums1[ik-1]`
- **Multiplied by** the **minimum** value of the selected elements from `nums2`: `min(nums2[i0], nums2[i1], ..., nums2[ik-1])`

Return the **maximum** possible score.

**Constraints:**
- `n == nums1.length == nums2.length`
- `1 <= n <= 10^5`
- `0 <= nums1[i], nums2[i] <= 10^5`
- `1 <= k <= n`

<Callout type="amber" label="Key Constraint">
  n can reach 10^5 and k can equal n, so an O(n²) or brute-force subsequence approach is infeasible. The sort + O(n log k) heap approach is required.
</Callout>

<Callout type="green" label="Rule of Thumb">
  When the score is a sum multiplied by a minimum, sort by the minimum descending and maintain the top-k sum with a min-heap — the current element's value IS the minimum for every valid selection that includes it.
</Callout>

<SectionHeader n={4} title="Approach" />

<Callout type="cyan" label="Core Insight">
  By sorting pairs descending by nums2, when we process pair i, nums2[i] is the minimum multiplier for any selection containing index i. A min-heap of size k tracks the top-k nums1 values seen so far, letting us compute the best sum for that multiplier in O(log k) per step.
</Callout>

Zip `nums1` and `nums2` into pairs. Sort descending by `nums2`. Iterate the sorted pairs:
- Add `nums1[i]` to the sum and push it into a min-heap.
- If the heap exceeds size `k`, pop the minimum — this removes the smallest nums1 value from consideration and updates the sum.
- When heap size equals `k`, update `score = max(score, sum * nums2[i])`.

At each index `i` (in sorted order), `nums2[i]` is the minimum multiplier for any selection that includes this index. The heap ensures we always keep the largest `k` values of `nums1` seen so far.

<Callout type="cyan" label="Why sort by nums2 descending">
By processing pairs from largest to smallest nums2, we guarantee that when we compute sum * nums2[i], the value nums2[i] is the minimum of everything in the heap (all previously considered pairs had larger nums2). This lets us enumerate every possible minimum exactly once.
</Callout>

<SectionHeader n={6} title="Step-by-Step Solution" />

<StepList>
  <StepItem title="Zip and sort">
    Sort `zip(nums1, nums2)` descending by the nums2 component (`.1`) to get pairs in the right processing order.
  </StepItem>
  <StepItem title="Iterate with a min-heap">
    For each `(n1, n2)` in pairs: `sum += n1`, `heap.insert(n1)`. If `heap.count > k`, pop min and subtract from sum.
  </StepItem>
  <StepItem title="Update score">
    When `heap.count == k`: `score = max(score, sum * n2)`.
  </StepItem>
  <StepItem title="Return score">
    Return `score`.
  </StepItem>
</StepList>

<SectionHeader n={7} title="Code Implementation" />

<Callout type="green" label="Swift Notes">
  The `Heap` from Swift Collections is used as a min-heap. `zip(nums1, nums2).sorted(by:)` with comparator `$0.1 > $1.1` creates `(n1, n2)` pairs sorted by n2 descending — `.1` accesses the second element of each tuple. `heap.popMin() ?? 0` handles the nil case safely (never triggered since `heap.count > k` guarantees a non-empty heap at that point).
</Callout>

```swift
import Collections

class Solution {
  func maxScore(_ nums1: [Int], _ nums2: [Int], _ k: Int) -> Int {
    let pairs = zip(nums1, nums2).sorted { $0.1 > $1.1 }
    var heap = Heap<Int>()   // min-heap
    var sum = 0
    var score = 0

    for (n1, n2) in pairs {
      sum += n1
      heap.insert(n1)

      if heap.count > k {
        sum -= heap.popMin() ?? 0
      }

      if heap.count == k {
        score = max(score, sum * n2)
      }
    }

    return score
  }
}
```

<SectionHeader n={8} title="Complexity Analysis" />

<Callout type="cyan" label="Time Complexity — O(n log n)">
  Sorting the n pairs is O(n log n). Each of the n iterations does a heap insert and at most one heap pop, each O(log k). Since k ≤ n, log k ≤ log n, so the loop is O(n log n). Total: O(n log n).
</Callout>

<Callout type="green" label="Space Complexity — O(n + k)">
  The sorted pairs array takes O(n). The heap holds at most k + 1 elements before the pop reduces it back to k — O(k). Total: O(n + k). Since k ≤ n this simplifies to O(n).
</Callout>
