<SectionHeader n={1} title="What is a Queue?" />

A queue enforces ordering: the first element added is the first one removed. This FIFO discipline is the natural model for level-order traversal (process all nodes at depth d before any at depth d+1) and for "recent calls" / sliding-window-count problems.

**Key properties:**

| Property | Detail |
|---|---|
| **Enqueue (append)** | Add to rear — O(1) amortised with Swift `Array.append` |
| **Dequeue (removeFirst)** | Remove from front — O(n) for `Array`, O(1) for linked-list or `Deque` |
| **Peek** | Inspect front without removing — O(1) |
| **FIFO guarantee** | Elements leave in exactly the order they entered |

<Callout type="amber" label="Key Constraint">
  Swift's built-in `Array.removeFirst()` is O(n) because it shifts all remaining elements. For problems requiring many dequeues (BFS, sliding window), use an **index pointer** instead of removing (`var front = 0; front += 1`) or use `Collections.Deque` from Swift Collections for true O(1) dequeue.
</Callout>

<SectionHeader n={2} title="Terminology" />

| Term | Definition |
|---|---|
| **Enqueue** | Add an element to the rear of the queue |
| **Dequeue** | Remove and return the element at the front |
| **Peek / front** | Inspect the front element without removing it |
| **FIFO** | First In, First Out — the defining property of a queue |
| **Deque** | Double-ended queue — supports O(1) insert/remove at both ends |
| **Monotonic deque** | A deque where elements are kept in sorted order; used for sliding window maximum |

<Callout type="green" label="Rule of Thumb">
  Use a queue (or BFS) whenever the problem processes items in arrival order or level by level. Use a **monotonic deque** when you additionally need to track the maximum or minimum within a sliding window.
</Callout>
