Populating Next Right Pointers in Each Node

#116MediumLevel-by-Level TraversalLeetCode ↗

Populating Next Right Pointers in Each Node

Linked ListTreeDepth-First SearchBreadth-First SearchBinary Tree
01

Problem Overview

You are given a perfect binary tree where all leaves are on the same level and every parent has two children. The tree node has an extra pointer next that should be populated to point to its next right node. If there is no next right node, next should be set to null.

Initially all next pointers are set to null. Populate every next pointer to point to its next right node on the same level.

Constraints:

  • The number of nodes in the tree is in the range [0, 2^12 - 1] (a perfect binary tree).
  • -1000 <= Node.val <= 1000
  • Follow-up: You may only use constant extra space (the recursion stack does not count for this purpose).

Pattern Recognition

→ Pattern Decision Tree
?
Is the tree guaranteed to be perfect (all leaves at same level)?
✓ Yes — A perfect binary tree means every node either has exactly 2 children or is a leaf — we can safely use left.next = right.
?
Can we use O(1) extra space (no queue)?
✓ Yes — Once the current level's next pointers are set, we can walk the level to connect children using those same next pointers.
?
Does each level's connection depend on the previous level?
✓ Yes — To link children across subtrees (right child of node to left child of node.next), the parent-level next pointers must already exist.
Tap to reveal pattern →
04

Approach

Maintain a leftMost pointer that starts at the root and descends one level per outer loop iteration. An inner curr pointer walks the current level using curr.next:

  • curr.left.next = curr.right — siblings within the same parent.
  • curr.right.next = curr.next.left — cross-parent link using the already-populated curr.next.
Why this is O(1) space

We never allocate a queue. The next pointers we set on the current level become the traversal links for the next iteration of the outer loop.

BFS fallback

A BFS solution with a queue is also valid and simpler to reason about, but uses O(n) space. The O(1) solution is preferred for this problem.

06

Step-by-Step Solution

Guard empty tree —

Return nil immediately if root is nil.

Outer loop: descend levels —

var leftMost = root; loop while leftMost.left != nil (not yet at the leaf level).

Inner loop: connect children —

var curr = leftMost; loop while curr != nil: set curr.left.next = curr.right, and if curr.next exists, set curr.right.next = curr.next.left. Advance curr = curr.next.

Descend to next level —

After the inner loop finishes, leftMost = leftMost.left to move down one level.

07

Code Implementation

class Solution {
  func connect(_ root: Node?) -> Node? {
    guard let root else { return nil }

    var leftMost: Node? = root

    while leftMost?.left != nil {
      var curr = leftMost

      while curr != nil {
        curr?.left?.next = curr?.right

        if let next = curr?.next {
          curr?.right?.next = next.left
        }

        curr = curr?.next
      }

      leftMost = leftMost?.left
    }

    return root
  }
}

09

Common Pitfalls

✕01
Applying this to an imperfect binary tree
The cross-parent link curr.right.next = curr.next.left assumes every node has two children. For an imperfect tree, curr.next.left may be nil when the actual next node exists elsewhere.
Fix: For problem 117 (imperfect tree), a different approach is needed. This solution is valid only for problem 116 (perfect binary tree guarantee).
✕02
Forgetting to advance curr after the inner loop body
Without curr = curr.next, the inner loop spins forever on the first node of each level.
Fix: Always advance curr = curr.next at the end of the inner while body.
✕03
Using leftMost.left instead of leftMost.left != nil as the loop condition
If you check leftMost != nil, the inner loop on a leaf level tries to access leftMost.left which is nil — no children to connect and no descent possible.
Fix: Check leftMost?.left != nil so the outer loop stops before trying to connect leaf-level children.
