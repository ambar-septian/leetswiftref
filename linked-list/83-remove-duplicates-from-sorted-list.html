Remove Duplicates from Sorted List

Linked List
01

Problem Overview

Given the head of a sorted linked list, delete all duplicates such that each element appears only once. Return the linked list sorted as well.

Unlike problem 82 (which removes all occurrences), this problem keeps one copy of each value and removes only the extra duplicates.

Constraints

  • The number of nodes in the list is in the range [0, 300].
  • -100 <= Node.val <= 100
  • The list is guaranteed to be sorted in ascending order.
Key Constraint

In a sorted list, duplicates of any value are always adjacent. The head is always kept — only the extra occurrences are removed. No dummy head is needed because the first node is never deleted.

Core Insight

In a sorted list, duplicates are always adjacent. Maintain a curr pointer. While curr.next has the same value as curr, skip curr.next by setting curr.next = curr.next.next (bypassing the duplicate). When the values differ, advance curr to curr.next.

Rule of Thumb

When duplicates are guaranteed to be adjacent (sorted list), a single pointer with an inner skip loop is sufficient — no hash set, no dummy head, no extra space needed.

04

Approach

Core Insight

Maintain a curr pointer. While curr.next has the same value as curr, bypass it by setting curr.next = curr.next.next. When values differ, advance curr to curr.next. No dummy head is needed — the first occurrence of every value is always kept.

Start with curr = head. For each node, use an inner pointer item = curr. While item.next exists and item.next.val == item.val, set item.next = item.next.next (bypass the duplicate). After the inner loop, advance curr = item.next.

The outer loop ensures every distinct value is processed; the inner loop eliminates all consecutive copies of the current value.

In-place re-linking vs. building a new list

Because we only keep one copy of each value and the first occurrence is always preserved, we can modify next pointers in place. No new nodes are created — O(1) extra space.

Step-by-Step Solution

Initialise pointers —

var curr = head, var result = curr.

Outer loop — advance through distinct values —

While curr != nil: use inner loop to skip duplicates of the current value.

Inner loop — bypass duplicate next nodes —

var item = curr. While item.next.val == item.val, set item.next = item.next.next.

Advance curr —

curr = item.next (the first node of the next distinct value).

07

Code Implementation

class Solution {
  func deleteDuplicates(_ head: ListNode?) -> ListNode? {
    var curr = head
    let result = curr

    while curr != nil {
      var item = curr
      while item?.val == item?.next?.val {
        item?.next = item?.next?.next   // bypass duplicate
      }
      curr = item?.next
    }

    return result
  }
}
Swift Notes

item?.next?.val uses optional chaining — when item?.next is nil, the entire expression short-circuits to nil, making the while condition item?.val == item?.next?.val evaluate to Optional(val) == nil, which is false, safely terminating the inner loop without a nil crash.

08

Complexity Analysis

Time Complexity — O(n)

O(n) — each node is visited at most twice: once by the outer loop advancing curr, and once by the inner loop bypassing a duplicate. Total work is proportional to the number of nodes.

Space Complexity — O(1)

O(1) auxiliary space. Re-linking next pointers in-place uses only a constant number of pointer variables (curr, item, result). No new nodes are created.

09

Common Pitfalls

✕01
Advancing curr inside the inner loop instead of after it
If curr is advanced during duplicate skipping, the outer loop may jump past the first occurrence of the next distinct value.
Fix: Use a separate item pointer for the inner loop. Advance curr only after the inner loop exits.
✕02
Confusing this problem with problem 82 (remove all duplicates)
Problem 83 keeps the first occurrence; problem 82 removes all occurrences. Using the problem-82 approach here would delete nodes that should be kept.
Fix: Only bypass next nodes whose value equals the current node's value — leave the current node in place.
✕03
Not handling nil next safely in Swift
item?.next?.val uses optional chaining. If item.next is nil, the comparison item?.val == item?.next?.val evaluates to Optional(val) == nil which is false — the inner loop exits safely.
Fix: Optional chaining handles nil termination correctly. No explicit nil guard is needed inside the inner loop.
