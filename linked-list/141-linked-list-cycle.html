Linked List Cycle

Hash TableLinked ListTwo Pointers
01

Problem Overview

Given head, the head of a linked list, determine if the linked list has a cycle in it.

There is a cycle in a linked list if there is some node in the list that can be reached again by continuously following the next pointer. Internally, pos is used to denote the index of the node that the tail's next pointer is connected to (0-indexed). Note that pos is not passed as a parameter.

Return true if there is a cycle in the linked list. Otherwise, return false.

Constraints:

  • The number of nodes in the list is in the range [0, 10^4].
  • -10^5 <= Node.val <= 10^5
  • pos is -1 (no cycle) or a valid index in the linked list.
  • Follow-up: Can you solve it using O(1) memory?

Pattern Recognition

Do we need O(1) space?
✓ Yes — A hash set approach works but uses O(n) space. Floyd's algorithm needs only two pointers.
?
Will a fast pointer eventually catch a slow pointer in a cycle?
✓ Yes — In a cycle, fast gains one step per iteration. Since the cycle is finite, it must lap slow.
?
Will fast reach nil if there is no cycle?
✓ Yes — Without a cycle, fast eventually hits nil — the loop terminates and we return false.

04

Approach

Initialise slow = head and fast = head.next. Each iteration: advance slow one step and fast two steps. If they ever point to the same node (===), a cycle exists. If fast or fast.next becomes nil, the list is acyclic.

Why start fast at head.next

Starting both at head would immediately satisfy slow === fast before any movement. Starting fast one step ahead avoids this false positive.

06

Step-by-Step Solution

Initialise pointers —

var slow = head, var fast = head?.next.

Loop while both non-nil —

while slow != nil, fast != nil.

Advance pointers —

slow = slow?.next, fast = fast?.next?.next.

Check for meeting —

After advancing, if slow === fast, return true.

No cycle —

Loop exits when fast or fast.next is nil — return false.

07

Code Implementation

class Solution {
  func hasCycle(_ head: ListNode?) -> Bool {
    var slow = head
    var fast = head?.next

    while slow != nil, fast != nil {
      slow = slow?.next
      fast = fast?.next?.next

      if slow === fast {
        return true
      }
    }

    return false
  }
}

09

Common Pitfalls

✕01
Using == instead of === for pointer comparison
Two different nodes can have the same val. Using value equality (==) would produce false positives on lists with duplicate values.
Fix: Use === (reference/identity equality) to check if slow and fast point to the exact same node object.
✕02
Starting both pointers at head
If slow and fast both start at head, the very first iteration check compares them before any movement, which would immediately return true for any non-empty list.
Fix: Start fast at head.next so the initial positions differ, then check after each movement.
✕03
Advancing fast before checking for nil
Calling fast.next.next without checking that fast.next is non-nil first causes a crash on acyclic lists with even length.
Fix: Swift's optional chaining handles this safely: fast?.next?.next returns nil if any link in the chain is nil.
