01

Problem Overview

Given the head of a singly linked list, reverse the list, and return the reversed list's head.

Constraints:

  • The number of nodes in the list is in the range [0, 5000].
  • -5000 <= Node.val <= 5000

Pattern Recognition

→ Pattern Decision Tree
?
Do we need to change the direction of every pointer?
✓ Yes — Each node's next must point to its predecessor — classic in-place pointer reversal.
?
Do we need extra memory proportional to the list length?
✗ No — Three local variables (curr, prev, temp) are all that is required — O(1) space.
?
Can we do it in a single pass?
✓ Yes — Each node is visited exactly once; prev accumulates the reversed chain as we advance.
Tap to reveal pattern →
04

Approach

Maintain two pointers: curr (starts at head) and prev (starts as nil). At each step:

  1. Save temp = curr.next before overwriting it.
  2. Redirect curr.next = prev (reverse the pointer).
  3. Advance prev = curr, then curr = temp.

When curr becomes nil, prev holds the new head.

Why save next first

Once we set curr.next = prev, the original next node is unreachable unless we saved it in temp. Always capture temp before the redirect.

06

Step-by-Step Solution

Initialise pointers —

var curr = head, var prev: ListNode? = nil.

Loop until curr is nil —

While curr != nil: save let temp = curr?.next.

Redirect pointer —

Set curr?.next = prev to reverse the arrow.

Advance both pointers —

prev = curr, then curr = temp.

Return new head —

Return prev — it points to the last node visited, which is the new head.

07

Code Implementation

class Solution {
  func reverseList(_ head: ListNode?) -> ListNode? {
    var curr = head
    var prev: ListNode?

    while curr != nil {
      let temp = curr?.next
      curr?.next = prev
      prev = curr
      curr = temp
    }

    return prev
  }
}

09

Common Pitfalls

✕01
Forgetting to save curr.next before redirecting
Setting curr.next = prev before saving curr.next in temp permanently cuts off the rest of the list.
Fix: Always capture let temp = curr?.next as the very first operation inside the loop body.
✕02
Returning curr instead of prev
When the loop ends, curr is nil. The new head is the last non-nil node visited, which is prev.
Fix: Return prev, not curr.
✕03
Not initialising prev to nil
The original tail of the list must point to nil after reversal. If prev starts as anything other than nil, the tail is incorrectly linked.
Fix: Declare var prev: ListNode? = nil before the loop.
