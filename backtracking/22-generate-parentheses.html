Generate Parentheses

StringDynamic ProgrammingBacktracking
01

Problem Overview

Given n pairs of parentheses, write a function to generate all combinations of well-formed parentheses.

A string of parentheses is well-formed if every opening bracket has a corresponding closing bracket and they are properly nested.

Constraints

  • 1 <= n <= 8
Key Constraint

n is at most 8, bounding the output to the 8th Catalan number (1430 strings). Every valid string has exactly 2n characters — both the open and close counts must reach exactly n.

Core Insight

At any point in building the string, you may add ( only if the open count is below n, and ) only if the close count is below the open count. These two rules together guarantee every candidate is valid — no pruning of invalid strings needed after the fact.

Rule of Thumb

When generating all well-formed bracket sequences, think constrained backtracking — add ( only when open count is below n, and ) only when close count is below open count. These two guards make every explored path valid by construction.

04

Approach

Core Insight

Two simple rules replace the need to validate strings after the fact: add ( only when open count is below n, and ) only when close count is below open count. These invariants guarantee every path in the recursion tree is already a valid prefix — no pruning of invalid strings is ever needed.

Use a recursive dfs(open, close) where open and close track how many of each bracket have been added so far. Two branches exist at each call: add ( if open < n, and add ) if close < open. When both counts equal n, the path is a complete valid string — append it to results and return.

Why close < open is the right guard

A ) is only valid if there is an unmatched ( to the left of it. The number of unmatched opens at any point is exactly open - close. Requiring close < open ensures we never add a ) without a pending ( to match.

06

Step-by-Step Solution

Initialise results and path —

results = [String](), path = [Character]() with capacity n * 2.

Define dfs(open, close) —

Base case: open == n && close == n — append String(path) to results and return.

Open bracket branch —

If open < n: append (, recurse with dfs(open+1, close), remove last.

Close bracket branch —

If close < open: append ), recurse with dfs(open, close+1), remove last.

Kick off and return —

Call dfs(0, 0). Return results.

07

Code Implementation

class Solution {
  func generateParenthesis(_ n: Int) -> [String] {
    var results = [String]()
    var path = [Character]()
    path.reserveCapacity(n * 2)

    func dfs(_ o: Int, _ c: Int) {
      if c == n && o == n {
        results.append(String(path))
        return
      }
      if o < n {
        path.append("(")
        dfs(o + 1, c)
        path.removeLast()
      }
      if c < o {
        path.append(")")
        dfs(o, c + 1)
        path.removeLast()
      }
    }

    dfs(0, 0)
    return results
  }
}
Swift Notes

path.reserveCapacity(n * 2) pre-allocates backing storage so no reallocation occurs during backtracking — a small but worthwhile optimisation when the final path length is known. The nested func dfs captures path, results, and n by reference automatically, no explicit capture list needed.

08

Complexity Analysis

Time Complexity — O(4^n / √n)

The number of valid strings is the nth Catalan number, which grows as O(4^n / n^(3/2)). Building each string takes O(n). Total time is O(4^n / √n * n).

Space Complexity — O(n)

The recursion depth is at most 2n (the length of one complete string). The path array holds at most 2n characters. Excluding the output array, auxiliary space is O(n).

When to prefer Backtracking over Brute Force

Always prefer backtracking — it generates only valid strings by construction, visiting roughly Catalan(n) leaves. Brute force generates all 2^(2n) binary strings and filters, which for n=8 means 65536 candidates reduced to 1430 valid ones — about 45x more work.

09

Common Pitfalls

✕01
Using close <= open instead of close < open
When close == open, all opens are matched — adding ) now would create a negative balance. The guard must be strict.
Fix: Use c < o (strictly less than), not c <= o.
✕02
Checking the base case with path.count instead of open == n && close == n
path.count == n * 2 is equivalent here, but open == n && close == n makes the invariant explicit and avoids a multiplication.
Fix: Either works. The solution uses c == n && o == n for clarity.
✕03
Forgetting removeLast after each recursive branch
Without path.removeLast() the path keeps accumulating characters from previous branches, producing garbled results.
Fix: Always call path.removeLast() immediately after each recursive dfs call returns.
