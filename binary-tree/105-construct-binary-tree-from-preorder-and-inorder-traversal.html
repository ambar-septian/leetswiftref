Construct Binary Tree from Preorder and Inorder Traversal

Construct Binary Tree from Preorder and Inorder Traversal

ArrayHash TableDivide and ConquerTreeBinary Tree
01

Problem Overview

Given two integer arrays preorder and inorder where preorder is the preorder traversal of a binary tree and inorder is the inorder traversal of the same tree, construct and return the binary tree.

  • Preorder traversal visits: root → left subtree → right subtree
  • Inorder traversal visits: left subtree → root → right subtree

These two traversals together uniquely determine a binary tree (assuming all values are distinct).

Constraints

  • 1 <= preorder.length <= 3000
  • inorder.length == preorder.length
  • -3000 <= preorder[i], inorder[i] <= 3000
  • preorder and inorder consist of unique values.
  • Each value of inorder also appears in preorder.
  • preorder is guaranteed to be the preorder traversal of the tree.
  • inorder is guaranteed to be the inorder traversal of the tree.
Key Constraint

All values are unique — this is what makes reconstruction possible. Without uniqueness, the root's position in the inorder array would be ambiguous. Preorder and inorder together uniquely determine the binary tree when values are distinct.

Core Insight

preorder[0] is always the root. Find that value's position (mid) in inorder — everything left of mid is the left subtree and everything right is the right subtree. A shared preIndex advances through preorder as each root is consumed; a hash map gives O(1) inorder lookups.

Rule of Thumb

Whenever you are given two traversals and asked to reconstruct the tree, use the first-element-is-root property of preorder plus the split-at-root property of inorder. Build a hash map for O(1) inorder lookups to avoid O(n²) total.

04

Approach

Core Insight

preorder[preIndex] is always the current root. Its position (mid) in inorder splits the array: inorder[inLeft..mid-1] is the left subtree and inorder[mid+1..inRight] is the right. A hash map gives O(1) lookups; a shared preIndex advances globally as each root is consumed — no array slicing needed.

Build hash[inorder[i]] = i for all i. Use a shared preIndex = 0 variable. Define recursive(inLeft, inRight):

  1. If inLeft > inRight return nil.
  2. Root value = preorder[preIndex]; increment preIndex.
  3. mid = hash[rootVal].
  4. root.left = recursive(inLeft, mid - 1), root.right = recursive(mid + 1, inRight).

06

Step-by-Step Solution

Build inorder hash map —

var hash = [Int: Int](). For i in 0..<inorder.count: hash[inorder[i]] = i. O(n) preprocessing.

Shared preIndex —

var preIndex = 0. This advances globally as each root is consumed — no need to pass subarray slices.

Recursive build —

recursive(inLeft, inRight): base case inLeft > inRight → nil. Consume preorder[preIndex++] as root. Split at mid = hash[rootVal]. Recurse left, then right.

07

Code Implementation

class Solution {
  func buildTree(_ preorder: [Int], _ inorder: [Int]) -> TreeNode? {
    var hash = [Int: Int]()
    for i in 0..<inorder.count { hash[inorder[i]] = i }

    var preIndex = 0

    func recursive(_ inLeft: Int, _ inRight: Int) -> TreeNode? {
      if inLeft > inRight { return nil }

      let rootVal = preorder[preIndex]
      preIndex += 1
      let root = TreeNode(rootVal)

      let mid = hash[rootVal]!
      root.left  = recursive(inLeft,    mid - 1)
      root.right = recursive(mid + 1,   inRight)

      return root
    }

    return recursive(0, inorder.count - 1)
  }
}
Swift Notes

var preIndex = 0 captured by the nested recursive closure is mutated with preIndex += 1 — Swift closures capture variables by reference, so each recursive call shares and advances the same counter without needing inout parameters. hash[rootVal]! force-unwraps safely because the problem guarantees all preorder values appear in inorder.

08

Complexity Analysis

Time Complexity — O(n)

O(n) for the hash-map approach. Building the hash map is O(n). Each of the n recursive calls does O(1) work (hash lookup, node creation, preIndex increment), giving O(n) total. The naive linear-search approach is O(n²) because each call scans up to n elements.

Space Complexity — O(n)

O(n) for the hash map storing all n inorder positions. The recursion stack is O(h) — O(n) for a skewed tree, O(log n) for a balanced tree. The output tree itself is O(n) nodes but is the required result.

When to prefer naive vs hash map

Always use the hash map + shared preIndex approach. The naive O(n²) solution is only acceptable for n ≤ 100 or as a conceptual starting point. For n = 3000 (the constraint limit), O(n²) is 9 million operations versus O(n) = 3000 — the hash map is strictly better.

09

Common Pitfalls

✕01
Recursing right before left
Preorder lists the left subtree values before the right subtree values. If you recurse right first, preIndex consumes right-subtree values for the left child — producing the wrong tree.
Fix: Always recurse left first: root.left = recursive(inLeft, mid - 1) before root.right = recursive(mid + 1, inRight).
✕02
Slicing arrays instead of using index bounds
Creating Array(preorder[1...mid]) at each level copies subarrays, making the total time O(n²).
Fix: Use (inLeft, inRight) bounds on the inorder array and a shared preIndex to avoid any array copying.
✕03
Off-by-one on left subtree bounds
The root is at inorder[mid]. The left subtree spans inorder[inLeft...mid-1]. Passing mid instead of mid-1 as the right bound includes the root in the left subtree recursion.
Fix: Left call: recursive(inLeft, mid - 1). Right call: recursive(mid + 1, inRight).
