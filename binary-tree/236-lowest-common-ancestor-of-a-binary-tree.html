Lowest Common Ancestor of a Binary Tree

#236MediumPost-Order DFSLeetCode ↗

Lowest Common Ancestor of a Binary Tree

TreeDepth-First SearchBinary Tree
01

Problem Overview

Given a binary tree, find the lowest common ancestor (LCA) of two given nodes p and q.

The lowest common ancestor is defined as the lowest node in the tree that has both p and q as descendants (where a node is allowed to be a descendant of itself).

Constraints:

  • The number of nodes in the tree is in the range [2, 10^5].
  • -10^9 <= Node.val <= 10^9
  • All node values are unique.
  • p != q
  • Both p and q exist in the tree.
Key Constraint

Both p and q are guaranteed to exist in the tree. This means the DFS is guaranteed to find both — no "not found" fallback is needed, and a node matching p or q can be returned immediately without searching its subtree.

02

Examples

Example 1​

Input: root = [3,5,1,6,2,0,8,null,null,7,4], p = 5, q = 1
Output: 3
Explanation: LCA of nodes 5 and 1 is 3.

Example 2​

Input: root = [3,5,1,6,2,0,8,null,null,7,4], p = 5, q = 4
Output: 5
Explanation: LCA of nodes 5 and 4 is 5, since a node can be a descendant of itself.

Rule of Thumb

When finding the LCA in a binary tree (not a BST), use post-order DFS — you need results from both subtrees before deciding at the current node. A node where both left and right return non-nil is the LCA.

04

Approach

Core Insight

Post-order DFS: return nil for a nil node, return root immediately if root === p or root === q. After recursing left and right: if both sides return non-nil, the current node is the split point — the LCA. If only one side is non-nil, propagate it upward via left ?? right.

Back at root 3: left=5 (non-nil), right=1 (non-nil)
Both subtrees returned non-nil → root 3 is the LCA
Key insight:
If left AND right both non-nil → current node is LCA.
If only one is non-nil → propagate that result up.

Recurse post-order. At each node:

  • If nil, return nil.
  • If root === p or root === q, return root (found one of the targets).
  • Recurse left and right. If both return non-nil, root is the LCA — return it. Otherwise return whichever side is non-nil (propagate upward).
Why returning root when root === p works

The problem guarantees both p and q exist in the tree. If root === p, then q must be somewhere in p's subtree or above. Either way, p is the LCA (a node can be its own ancestor). Returning root early is correct.

06

Step-by-Step Solution

Base case — nil node —

Return nil immediately if root is nil.

Base case — target found —

If root === p or root === q, return root. This covers the case where a node is its own ancestor.

Post-order recurse —

let left = lca(root.left, p, q), let right = lca(root.right, p, q).

Both non-nil — LCA found —

If left != nil && right != nil, return root.

Propagate non-nil side —

Otherwise return left ?? right — pass the found node up the call stack.

07

Code Implementation

class Solution {
  func lowestCommonAncestor(_ root: TreeNode?, _ p: TreeNode?, _ q: TreeNode?) -> TreeNode? {
    guard let root else { return nil }

    if root === p || root === q { return root }

    let left  = lowestCommonAncestor(root.left,  p, q)
    let right = lowestCommonAncestor(root.right, p, q)

    if left != nil, right != nil { return root }
    return left ?? right
  }
}
Swift Notes

root === p uses reference equality (===) to test that two variables point to the same object — semantically correct for node identity. left ?? right cleanly returns the non-nil side without an explicit if-else. The function signature takes TreeNode? optionals matching LeetCode's generated Swift template.

08

Complexity Analysis

Time Complexity — O(n)

Every node is visited at most once. The early return when root === p or root === q skips subtree traversal for the found node, but in the worst case (LCA at root) all nodes are visited.

Space Complexity — O(h)

The call stack depth equals tree height h. O(h) = O(log n) for balanced, O(n) for a skewed or path-like tree.

09

Common Pitfalls

✕01
Not returning early when root matches p or q
Without the early return, the DFS would recurse into p's subtree unnecessarily. Worse, if q is in p's subtree, the code would return q instead of p as the LCA.
Fix: Always return root immediately when root === p || root === q, before recursing.
✕02
Returning root when only one side is non-nil
If only the left subtree finds a target, returning root instead of left claims the current node is the LCA, which is wrong.
Fix: Only return root when BOTH left and right are non-nil. Otherwise return left ?? right.
✕03
Using value equality (==) instead of reference equality (===)
The problem guarantees unique values, so value equality would work here — but using === is semantically correct for node identity and safer in general.
Fix: Use root === p (reference equality) rather than root?.val == p?.val.
