Accounts Merge

01

Problem Overview

Given a list of accounts where accounts[i] is a list of strings such that accounts[i][0] is a name and the remaining elements are emails for that account, merge accounts that share at least one email in common.

Two accounts definitely belong to the same person if there is some common email. Note that even if two accounts have the same name, they may belong to different people since a single person can have multiple names. A person can have any number of accounts initially, but all accounts will eventually have the same name.

After merging the accounts, return the accounts in the following format: the first element of each account is the name, and the rest of the elements are emails in sorted order. The accounts themselves can be returned in any order.

Constraints:

  • 1 <= accounts.length <= 1000
  • 2 <= accounts[i].length <= 10
  • 1 <= accounts[i][j].length <= 30
  • accounts[i][0] consists of English letters.
  • accounts[i][j] (for j > 0) is a valid email.

04

Approach

Phase 1 — Build union-find on account indices: for each account, scan its emails. If an email has been seen before (it is in emailToAccIndex), unite(currentAccountIndex, storedIndex). Otherwise record emailToAccIndex[email] = currentAccountIndex.

Phase 2 — Collect: for every email in the map, find its root unionFind.find(storedIndex) and append the email to emailGroup[root].

Phase 3 — Build result: for each root, prepend the account name (accounts[root][0]), sort emails, and append to results.

Why index accounts, not emails?

Union-Find operates on indices. By assigning each account an index and uniting account indices when their emails overlap, we avoid creating a Union-Find structure over variable-length email strings. The email map acts as the bridge between emails and account indices.

Step-by-Step Solution

Initialise Union-Find —

UnionFind(accounts.count) — parent and rank arrays of size n.

Phase 1 — Map emails and unite —

For each (i, account): for each email in account[1...]: if already in emailToAccIndex, unite(i, emailToAccIndex[email]); else emailToAccIndex[email] = i.

Phase 2 — Group emails by root —

For each (email, i) in emailToAccIndex: let root = find(i). Append email to emailGroup[root].

Phase 3 — Build result —

For each (root, emails) in emailGroup: prepend accounts[root][0] (the name), sort emails, append to results.

07

Code Implementation

class Solution {
  class UnionFind {
    var parent = [Int]()
    var rank = [Int]()

    init(_ n: Int) {
      parent = (0..<n).map { $0 }
      rank = Array(repeating: 0, count: n)
    }

    func find(_ i: Int) -> Int {
      if parent[i] == i { return i }
      parent[i] = find(parent[i])
      return parent[i]
    }

    func unite(_ a: Int, _ b: Int) -> Bool {
      let rootA = find(a), rootB = find(b)
      guard rootA != rootB else { return false }
      if rank[rootA] < rank[rootB] { parent[rootA] = rootB }
      else if rank[rootA] > rank[rootB] { parent[rootB] = rootA }
      else { rank[rootB] += 1; parent[rootA] = rootB }
      return true
    }
  }

  func accountsMerge(_ accounts: [[String]]) -> [[String]] {
    var emailToAccIndex = [String: Int]()
    let uf = UnionFind(accounts.count)

    for (i, account) in accounts.enumerated() {
      for j in 1..<account.count {
        let email = account[j]
        if let existing = emailToAccIndex[email] {
          uf.unite(i, existing)
        } else {
          emailToAccIndex[email] = i
        }
      }
    }

    var emailGroup = [Int: [String]]()
    for (email, i) in emailToAccIndex {
      let root = uf.find(i)
      emailGroup[root, default: []].append(email)
    }

    var results = [[String]]()
    for (root, emails) in emailGroup {
      results.append([accounts[root][0]] + emails.sorted())
    }
    return results
  }
}

09

Common Pitfalls

✕01
Uniting with the wrong index
You must store the account index (not the email) in emailToAccIndex. Uniting email strings directly is not supported by array-based Union-Find.
Fix: emailToAccIndex maps email → account index. Call unite(currentAccountIndex, storedIndex) when a collision is detected.
✕02
Using root as the name source
After grouping, the root index points to one of the original accounts. Using accounts[root][0] gives the correct name for that merged group.
Fix: Access accounts[root][0] for the name — root is always one of the original account indices, so its name is valid.
✕03
Forgetting to sort emails
The problem requires emails in sorted order within each merged account.
Fix: Call emails.sorted() before appending to the result.
