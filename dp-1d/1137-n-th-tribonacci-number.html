N-th Tribonacci Number

#1137EasyDynamic Programming (1D) — MemoizationLeetCode ↗

N-th Tribonacci Number

MathDynamic ProgrammingMemoization
01

Problem Overview

The Tribonacci sequence T(n) is defined as follows:

  • T(0) = 0
  • T(1) = 1
  • T(2) = 1
  • T(n+3) = T(n) + T(n+1) + T(n+2) for n >= 0

Given n, return the value of T(n).

Constraints:

  • 0 <= n <= 37
  • The answer is guaranteed to fit within a 32-bit integer, i.e., answer <= 2^31 - 1.
Key Constraint

Base cases are T(0) = 0, T(1) = 1, T(2) = 1. Without explicit handling of all three, the recursion bottoms out incorrectly. n can be 0 — return 0 immediately for that case.

02

Examples

Example 1​

Input:  n = 4
Output: 4
Explanation: T(4) = T(1) + T(2) + T(3) = 1 + 1 + 2 = 4

Output: 1389537

Rule of Thumb

When a recurrence depends only on the last k values, you need only k variables — not a full array. For Tribonacci, three rolling variables replace an O(n) dp array, reducing space to O(1).

04

Approach

Core Insight

Tribonacci is a pure DP recurrence: T(n) = T(n-1) + T(n-2) + T(n-3). Top-down memoization stores each subproblem result once, reducing exponential recursion to O(n). The bottom-up rolling-window variant eliminates the cache entirely by keeping only the last 3 values — O(1) space.

The recursive structure mirrors Fibonacci but sums three predecessors. A top-down approach with a dictionary cache eliminates redundant computation: once T(k) is computed, it is stored and never recomputed.

An alternative bottom-up approach maintains three variables a, b, c and rolls them forward, using O(1) space. Both are correct; the top-down approach maps directly to the recurrence definition.

Base cases

T(0) = 0, T(1) = 1, T(2) = 1. In the Swift implementation: n less than or equal to 0 returns 0 and n less than 2 returns 1 (covering n=1). For n=2, the recursion computes T(-1)+T(0)+T(1) = 0+0+1 = 1, which is correct since T(-1) hits the n less than or equal to 0 guard.

06

Step-by-Step Solution

Check cache —

If cache[n] exists, return it immediately — no further work needed.

Handle base cases —

Return 0 if n less than or equal to 0, return 1 if n less than 2 (covers n=1).

Recurse and cache —

Compute tribonacci(n-3) + tribonacci(n-2) + tribonacci(n-1), store in cache[n], and return it.

07

Code Implementation

class Solution {
  var cache = [Int: Int]()

  func tribonacci(_ n: Int) -> Int {
    if let c = cache[n] { return c }
    if n <= 0 { return 0 }
    if n < 2  { return 1 }
    let result = tribonacci(n - 3) + tribonacci(n - 2) + tribonacci(n - 1)
    cache[n] = result
    return result
  }
}
Swift Notes

Use var cache = [Int: Int]() as an instance property so it persists across the single call (the recursive calls share it). Check if let c = cache[n] { return c } before base-case guards to short-circuit immediately. For the rolling-window variant: (a, b, c) = (b, c, a + b + c) in Swift is a clean tuple swap — no temporary variable needed.

08

Complexity Analysis

Time Complexity — O(n)

With memoization, each subproblem T(k) is computed exactly once for k in 0..n. The cache lookup is O(1). Total: O(n) calls. The bottom-up rolling window is also O(n) with n iterations.

Space Complexity — O(n) memoization / O(1) rolling

Memoization stores n + 1 entries in the cache dictionary — O(n). The call stack adds another O(n) in the worst case. The bottom-up rolling-window variant uses only 3 variables — O(1) space.

Common Pitfalls

✕01
Wrong base case boundaries
Using n==0 and n==1 checks separately while missing n==2 causes infinite recursion or wrong results for small inputs.
Fix: Use n less than or equal to 0 → 0 and n less than 2 → 1. For n=2 the recursion itself computes T(-1)+T(0)+T(1)=0+0+1=1 correctly.
✕02
Forgetting to cache before returning
Without storing the result in the cache, the memoization provides no benefit and performance degrades to exponential.
Fix: Always set cache[n] = result before returning.
✕03
Using an instance-level cache across test cases
LeetCode may reuse the same Solution instance for multiple test cases. A stale cache from a previous call may return wrong results.
Fix: Initialise cache inside the function or reset it at the start of each call.
